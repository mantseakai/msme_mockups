How the GPS check works

Overview of the proximity verification flow

When a Field Agent opens Visit Log for an MSME, the app silently requests their current GPS position and computes the straight-line distance to the MSME's stored coordinates (captured at registration). If within 50 metres, the form unlocks automatically. No manual steps required from the agent.

Checking - form locked
GPS acquisition in progress. Form is disabled. Progress bar shown. Timeout after 15 seconds → escalates to GPS Failed state.
In range — form unlocks
Agent is within 50m of the MSME location. Exact distance shown. Form enables. GPS coords logged on the visit record.
Out of range — form blocked
GPS acquired but agent is >50m from MSME. Distance shown. Form stays locked. Agent must move closer. No override option — this is intentional.
GPS failed — override required
GPS timed out, permission denied, or signal lost. Agent must select an override reason from a fixed list. Visit is saved with gps_verified: false and flagged for supervisor review.

Flutter implementation

Step-by-step build guide

1
Add the geolocator package
Add to pubspec.yaml:
dependencies: geolocator: ^11.0.0
Add permissions to AndroidManifest.xml:
<uses-permission android:name= "android.permission.ACCESS_FINE_LOCATION"/> <uses-permission android:name= "android.permission.ACCESS_COARSE_LOCATION"/>
2
Haversine distance function
Use Geolocator's built-in method — no external dependency needed:
double distanceTo( double msme_lat, double msme_lng, double agent_lat, double agent_lng, ) { // Returns distance in metres return Geolocator.distanceBetween( agent_lat, agent_lng, msme_lat, msme_lng, ); } // Threshold — configurable per market const double PROXIMITY_LIMIT_M = 50.0;
3
GPS check on page load
Run this in initState() of the Visit Log screen:
enum GpsState { checking, success, outOfRange, failed } Future<void> runGpsCheck() async { setState(() => gpsState = GpsState.checking); // Check permission first final perm = await Geolocator.checkPermission(); if (perm == LocationPermission.denied) { await Geolocator.requestPermission(); } try { final pos = await Geolocator.getCurrentPosition( desiredAccuracy: LocationAccuracy.high, timeLimit: const Duration(seconds: 15), ); final dist = distanceTo( msme.gpsLat, msme.gpsLng, pos.latitude, pos.longitude, ); setState(() { agentLat = pos.latitude; agentLng = pos.longitude; distanceToMsme = dist; gpsState = dist <= PROXIMITY_LIMIT_M ? GpsState.success : GpsState.outOfRange; }); } on TimeoutException { setState(() => gpsState = GpsState.failed); } catch (_) { setState(() => gpsState = GpsState.failed); } }
4
Form enable/disable logic
Gate the form fields and submit button on GPS state:
bool get formEnabled => gpsState == GpsState.success || gpsState == GpsState.failed; // failed = override mode bool get canSubmit { if (gpsState == GpsState.success) { return visitTypeSelected && noteNotEmpty; } if (gpsState == GpsState.failed) { return visitTypeSelected && overrideReasonSelected; } return false; }
5
Override reason dropdown
Show this only in GPS Failed state. Use a fixed list — no free text:
const List<String> overrideReasons = [ 'Device location turned off', 'Indoor — no GPS signal', 'Market area GPS coverage gap', 'Device GPS hardware issue', 'Other', ];

Visit record — data model

Fields to store on every visit log submission

// POST /api/v1/visits — request body { "msme_id": "uuid", "field_agent_id": "FA-GAR-ACC-047", "visit_type": "onboarding | follow_up | support | audit", "note": "string | null", // GPS fields — always included "gps_verified": true | false, "agent_lat": 5.5502, "agent_lng": -0.2074, "gps_accuracy_m": 4, "distance_to_msme_m": 23, "gps_timestamp": "ISO 8601", // Only populated when gps_verified = false "override_reason": "Indoor — no GPS signal" | null }
Always store raw GPS coordinates and computed distance — not just the boolean. This lets supervisors map every visit and detect patterns (e.g. agent always logging from 400m away on "GPS failed").

Edge cases to handle

Build these before go-live

ScenarioBehaviourPriority
MSME has no stored GPS coordinates Skip proximity check. Log visit with gps_check: skipped_no_msme_coords. Do not block. Blocker
GPS permission denied by user Show permission prompt. If denied, fall through to GPS Failed state with override required. Blocker
GPS accuracy >50m (poor signal) Still check distance but show accuracy warning. If accuracy > threshold distance, treat as GPS Failed. High
Agent at edge of 50m boundary Apply the limit strictly. 51m = out of range. No "close enough" fuzzing — the fixed list override exists for genuine cases. Normal
Agent retries GPS (from failed state) Retry button re-runs the full check flow from scratch. Retry max 3 times before locking to override-only. High
Sunmi device location services off Detect in initState() via Geolocator.isLocationServiceEnabled(). Show prompt to enable. If user ignores, fall to GPS Failed. Blocker
Multiple MSMEs at same location Proximity check passes for all. Business dropdown is still required — agent must select the correct one. Normal

Backend validation

Re-verify on the server — never trust the client

The Flutter app enforces the GPS check, but the backend must independently recompute the distance and validate it. A tampered client could send gps_verified: true with fake coordinates.
# Django — visits/views.py from geopy.distance import geodesic PROXIMITY_LIMIT_M = 50 def validate_visit_gps(data, msme): if not msme.gps_lat or not msme.gps_lng: data['gps_check'] = 'skipped_no_msme_coords' data['gps_verified'] = False return data if data.get('agent_lat') is None: data['gps_verified'] = False return data dist = geodesic( (data['agent_lat'], data['agent_lng']), (msme.gps_lat, msme.gps_lng), ).meters # Override the client-submitted distance with server value data['distance_to_msme_m'] = int(dist) data['gps_verified'] = dist <= PROXIMITY_LIMIT_M # If client claims verified but server says no → flag it if not data['gps_verified'] and not data.get('override_reason'): raise ValidationError('GPS check failed — override reason required') return data
Install geopy on the backend: pip install geopy. Do not use a simple Euclidean distance — it is inaccurate at GPS scales near the equator.

Supervisor portal — what to show

Visibility into GPS-flagged visits

The portal's visit log list must surface GPS status clearly so supervisors can prioritise review.

BadgeMeaningSupervisor action
GPS verifiedAgent was within 50m at time of logNo action required
Override — reviewGPS failed, agent used manual overrideReview reason + map position
GPS mismatchServer distance >50m but client sent verified=trueEscalate — possible tampering
No MSME GPSMSME has no stored coordinatesAssign a Field Agent to capture GPS
Send supervisors a daily digest of override-flagged visits — not just a portal list. An agent with 3+ overrides in a week is a pattern, not noise. This can be a simple email summary job run each morning.
